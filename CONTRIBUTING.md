# Contributing to Nebula Killsay

Thanks for your interest in contributing. Here's how to do it well.

---

## Bug Reports

Use the **Bug Report** issue template. Before filing:

- Check [existing issues](https://github.com/yourusername/nebula-killsay/issues) — it may already be reported
- Make sure you're on the latest version
- Include your Windows version, Python version (if running from source), and CS2 build

Good bug reports include:
- What you expected to happen
- What actually happened
- Steps to reproduce reliably
- The contents of the log panel (copy via the Export Log button)

---

## Feature Requests

Use the **Feature Request** issue template. Be specific — "make it better" is hard to act on, "add support for `{deaths}` variable" is not.

Note that Pro-tier features will not be added to the free tier regardless of pull requests.

---

## Pull Requests

PRs are welcome for:
- Bug fixes
- Weapon name additions / corrections
- Documentation improvements
- UI polish (within the existing design system)

PRs will **not** be accepted for:
- Removing or bypassing license key checks
- Adding features that conflict with the Pro tier roadmap
- Changing the core message delivery mechanism
- Dependency changes without prior discussion

### Process

1. Fork the repo and create a branch: `git checkout -b fix/your-bug-name`
2. Make your changes
3. Test manually — load into a CS2 match and verify the fix
4. Submit a PR with a clear description of what changed and why

By submitting a PR you agree your contribution may be merged under the project's license without additional compensation.

---

## Code Style

- Follow the existing style — no reformatters, no linters introduced without discussion
- Keep it readable — this codebase prioritises clarity over cleverness
- Comments in English
